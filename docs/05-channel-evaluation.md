# 05 — Channel Evaluation

This chapter scores the five distribution channels a Cytnx maintainer could plausibly ship through — build-from-source, pip/wheels, conda(-forge), pixi, uv — against six explicit criteria. It draws only on findings already established in chapters 01 (current-state audit), 02 (dependency taxonomy), 03 (core problems), and 04 (reference-project patterns), grounding every matrix cell in those four chapters; no new source citations beyond chapters 01–04 are introduced here. Every matrix cell names the specific chapter/section it rests on. §5.4 states which channel(s) the matrix favors, as the input chapter 06 will turn into a recommendation and roadmap — this chapter does not itself propose an implementation plan.

## 5.1 Criteria

- **Dependency coverage** — how much of chapter 02's dependency set (CPU math: OpenBLAS/MKL/ARPACK; GPU stack: CUDA/cuTENSOR/cuQuantum; build-only: hptt/pybind11/Boost) the channel can actually deliver to a user, as a real installed artifact.
- **GPU support** — whether the channel ships (or could ship, given what already exists in this codebase) a `USE_CUDA=ON` build, not just whether GPU packages theoretically exist upstream.
- **Runtime-resolution robustness** — how well the channel avoids the import-time failure modes chapter 03 §3.2 catalogued (wrong lib found, LP64/ILP64 ABI mismatch, GPU driver/toolkit skew, pip-vs-conda conflict).
- **Reproducibility** — whether two installs of "the same" release, on two machines or two days apart, resolve to the same physical library artifacts (chapter 03 §3.1's stated reproducibility failure: "a preset name is not a pin on which physical library file each enabled option resolves to").
- **Maintainer burden** — how much new CI/packaging work the channel requires beyond what chapter 01 shows already exists.
- **User experience** — what a user actually has to type/install/know to get a working `import cytnx`.

## 5.3 Platform notes

**Linux + CUDA.**
- *build-from-source* is the only channel that can reach `USE_CUDA=ON`/`USE_CUTENSOR=ON`/`USE_CUQUANTUM=ON` today, via the `openblas-cuda`/`mkl-cuda` presets (ch.01 §1.1) — but this path has never once been exercised by CI (ch.01 §1.2), so "can reach" does not mean "known to work."
- *pip/wheels* builds manylinux/musllinux wheels on `ubuntu-24.04`/`ubuntu-24.04-arm` (ch.01 §1.2) — CPU only, no CUDA variant is built or planned in current workflows.
- *conda(-forge)* builds on `ubuntu-latest` (ch.01 §1.2) with no CUDA path in `conda_build_release.yml`; conda-forge itself already carries `cudatoolkit`/`cutensor`/`cuquantum` (ch.02 §2.1), which is real *potential* Cytnx's own recipe doesn't yet use.
- *pixi* inherits conda-forge's CUDA-package availability by construction but, like conda, has no existing Cytnx recipe that turns `USE_CUDA` on.
- *uv* has no CUDA story at all — identical to pip, since it consumes the same wheel index.

**macOS (arm64/Accelerate, no CUDA).**
- *build-from-source*: the `openblas-apple` preset exists (ch.01 §1.1) with LTO force-disabled specifically for macOS (`CMakeLists.txt:227-231`, ch.01 §1.1); `adv_install.rst` recommends MKL on Mac (because Homebrew OpenBLAS allegedly lacks `lapacke.h`) while the actual CI/wheel build links Homebrew OpenBLAS directly with no MKL fallback — a documented-vs-actual divergence (ch.02 §2.2). Apple's own Accelerate framework — free, OS-provided, needing no packaging step — is used by neither the docs' recommendation nor the actual build (ch.03 §3.1 "Platform angle"); MKL itself is not a native arm64 BLAS vendor, and nothing warns a user off selecting `mkl-cpu` on `arm64` (ch.03 §3.1).
- *pip/wheels*: wheels are built on `macos-14` (Apple Silicon) and `macos-15-intel` (ch.01 §1.2), with `delocate` bundling the Homebrew-installed OpenBLAS/ARPACK (ch.02 §2.2); CUDA is moot since NVIDIA dropped macOS CUDA support after version 10.2 (ch.02 §2.2).
- *conda(-forge)*: `conda_build_release.yml` also targets `macos-15-intel` and `macos-latest` (ch.01 §1.2), so arm64 is covered; conda-forge could in principle offer an Accelerate-linked OpenBLAS build the way SciPy exposes `-Dblas=accelerate` as an explicit, named build option (ch.04 §4.1 pattern S1) — Cytnx's own recipe does not currently do this.
- *pixi*: same conda-forge macOS-arm64 package ceiling as conda, same caveat that no Cytnx-specific recipe exists yet.
- *uv*: consumes whatever wheel PyPI publishes — identical macOS coverage to pip, since it's the same artifact.

**Windows.**
No channel works today. `conda_build/bld.bat`, the Windows conda build script, is a literal 0-line empty file (ch.01 §1.1/§1.2), and no `windows-*` GitHub Actions runner appears in any workflow (ch.01 §1.2) — Windows support is absent from CI entirely, not merely untested like the Linux GPU path. *build-from-source* is theoretically attemptable by a sufficiently motivated user with MSVC and manually-supplied dependencies, but is wholly unvalidated by anything in this repository. *pip/wheels* and *conda(-forge)* both simply have no Windows artifact to install. *pixi* and *uv* are downstream of those same artifacts — with no wheel and no conda package to consume, neither has anything Windows-specific to offer beyond what upstream chapters already show is missing.

## 5.4 Leading option

The matrix does not point to one channel that wins outright on every criterion — it points to a split between *what already works today* and *what the dependency shape structurally rewards*.

**pip/wheels is the strongest channel for immediate, low-friction CPU access.** It is the only channel besides conda that is actually CI-built and published today (ch.01 §1.2), its runtime-resolution story for the one configuration it covers is already solved by `cibuildwheel`'s default repair step (ch.02 §2.3(a), ch.03 §3.2), and its maintainer cost to keep working is near zero. Its ceiling, however, is hard: it cannot express the GPU stack at all without adopting genuinely new machinery (chapter 04's CuPy patterns C1–C3), and its robustness has one documented crack — the pip-vs-conda `LD_LIBRARY_PATH`/`DT_RUNPATH` conflict (ch.03 §3.2 failure mode 4). **uv rides on exactly this same ceiling and the same crack** — it adds resolver speed and a Python-level lockfile at essentially no extra maintainer cost, but changes nothing about what's inside the wheel.

**conda(-forge) and pixi score best on the two criteria the matrix weighs most heavily against Cytnx's actual dependency shape: runtime-resolution robustness and dependency coverage.** Chapter 02 §2.1 already shows that nearly every dependency in Cytnx's taxonomy — including the entire proprietary GPU stack (`cudatoolkit`, `cutensor`, `cuquantum`) that pip/wheels cannot touch at all — already exists as a conda-forge package. And chapter 03's central diagnosis (build-time discovery ambiguity propagating unrepaired into import-time fragility) is structurally the class of problem a real dependency-resolving package manager is built to prevent: conda's solver would replace ARPACK's name-only `find_library` (failure mode 1) and MKL's silent LP64/ILP64 default (failure mode 2) with declared, version-tracked package dependencies instead of ambient search-path luck. Between the two, pixi's lockfile (`pixi.lock`) offers a stronger reproducibility guarantee than plain conda-build's version-range `meta.yaml` (ch.02 §2.2's Boost host/run pin observation shows real but uneven rigor there) — but pixi starts from zero (no existing Cytnx manifest anywhere in chapter 01's findings), while conda-build at least has a working, CI-exercised recipe already in the repository, even though it currently ships to a personal channel rather than conda-forge itself (ch.01 §1.2).

Net implication for chapter 06: **the conda-forge-based ecosystem (conda and, on top of it, pixi) is where the matrix says the GPU-stack and runtime-robustness problems chapters 02–03 diagnosed are structurally addressable; pip/wheels-plus-uv is where the matrix says the lowest-cost, already-working CPU story lives.** Nothing in this chapter's evidence supports treating build-from-source as more than the necessary baseline every other channel is built on top of — it loses on robustness, reproducibility, and UX simultaneously, for reasons chapter 03 already established are structural rather than incidental.
