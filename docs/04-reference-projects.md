# 04 — Reference Projects

Chapter 03 named two problems: build-time discovery that is inconsistent across five different mechanisms (§3.1), and import-time resolution that is fragile because it inherits whatever ambiguity build-time discovery left behind (§3.2). This chapter asks how two mature, actively-shipped scientific-Python projects — SciPy (BLAS/LAPACK, a compiled-runtime-library problem structurally close to Cytnx's OpenBLAS/MKL/ARPACK tier, ch.02 §2.3(a)) and CuPy (CUDA/cuTENSOR, structurally close to Cytnx's CUDA/cuTENSOR/cuQuantum tier, ch.02 §2.3(b)) — solve exactly these problems in their own public build and packaging configuration. Every citation below is a URL to the live SciPy or CuPy repository or documentation, fetched and checked on 2026-07-03, not the Cytnx submodule. §4.3 maps each pattern extracted here back to a specific chapter-03 problem; no channel recommendation for Cytnx is made in this chapter — that is chapters 05–06's job. This chapter only distills mechanisms.

## 4.1 SciPy — the BLAS/LAPACK build-time selection and wheel-repair playbook

### Pattern S1 — build-time BLAS/LAPACK selection is an explicit, enumerated build option, not ambient discovery

SciPy's Meson build exposes BLAS/LAPACK choice as two named build options, `-Dblas=` and `-Dlapack=`, documented at `https://docs.scipy.org/doc/scipy/building/blas_lapack.html`. The page states the default is OpenBLAS, and enumerates the other values a packager can pass explicitly: plain Netlib libblas/liblapack (`-Dblas=blas -Dlapack=lapack`), Apple's Accelerate (`-Dblas=accelerate`, macOS 13.3+ only, added in SciPy 1.14.0), MKL (`-Dblas=mkl-dynamic-lp64-gomp` or an equivalent pkg-config name), BLIS (`-Dblas=blis`, paired with an explicit `-Dlapack=` since BLIS is BLAS-only), and ATLAS (`-Dblas=blas-atlas`). Discovery itself is deterministic and stated as policy, not left to whatever the compiler's default search path happens to contain: "Meson tries to discover the specified libraries first with `pkg-config`, and then with CMake." An additional flag, `-Duse-g77-abi=true`, exists specifically to override SciPy's automatic Fortran-ABI detection when linking plain libblas/liblapack — i.e., the one ABI hazard chapter 03 flagged for Cytnx's MKL LP64/ILP64 choice (§3.2 failure mode 2) has a direct SciPy analogue, and SciPy gives packagers an explicit override rather than silently assuming one convention. SciPy's `spin` developer-CLI wraps the common cases into named shortcuts, e.g. `spin build --with-accelerate` and `spin build --with-scipy-openblas=32` (same page).

The wheel-build CI does not rely on whatever OpenBLAS a runner's package manager happens to have installed. `tools/wheels/cibw_before_build.sh` (`https://github.com/scipy/scipy/blob/main/tools/wheels/cibw_before_build.sh`) installs a purpose-built, pip-installable OpenBLAS distribution — `scipy-openblas32` or `scipy-openblas64`, chosen by runner architecture (`RUNNER_ARCH`; Windows ARM64 forces the 32-bit variant) — via `python -m pip install -r requirements/openblas.txt`, then generates a `.pc` pkg-config file from that installed package so Meson's `pkg-config`-first discovery (S1, above) finds it deterministically inside the CI container. On Windows, the script additionally copies the resulting OpenBLAS DLLs to a fixed location so `delvewheel` (Windows' analogue of `auditwheel`/`delocate`) can find and bundle them. The `scipy-openblas32` package's own PyPI description (`https://pypi.org/project/scipy-openblas32/`) states its purpose explicitly: "to use them as build and runtime dependencies in CI and local development for NumPy and SciPy" and "to be vendored into NumPy and SciPy wheels" — and warns other projects off depending on it directly ("Please do not add a runtime dependency on these wheels if you're not NumPy or SciPy. This is not supported and likely to lead to breakage or symbol conflicts."). The mechanism this exposes: a wheel-building CI pipeline can pin its own reproducible, pip-installable copy of a compiled-runtime dependency, entirely separate from — and not vulnerable to drift from — whatever build of that same library a packager's host system or conda environment supplies.

### Pattern S2 — post-build wheel repair (auditwheel/delocate/delvewheel) vendors the linked libraries into the wheel, documented as deliberate policy

SciPy's own docs state the mechanism directly (`https://docs.scipy.org/doc/scipy/building/redistributable_binaries.html`): "run `auditwheel` (Linux), `delocate` (macOS), `delvewheel` (Windows) or `repairwheel` (platform-independent) to vendor the required shared libraries into the wheel," because an unrepaired wheel "will rely on external shared libraries (at least for BLAS/LAPACK and a Fortran compiler runtime library, perhaps other libraries)." SciPy's `pyproject.toml` (`https://github.com/scipy/scipy/blob/main/pyproject.toml`) confirms this is exactly what CI does: the `[tool.cibuildwheel]` table sets `before-build = "bash {project}/tools/wheels/cibw_before_build.sh {project}"` and, for Linux and macOS, sets no `repair-wheel-command` override at all — cibuildwheel's own documented default therefore runs (`auditwheel repair` on Linux, `delocate` on macOS); only Windows gets an explicit override, `repair-wheel-command = "bash ./tools/wheels/repair_windows.sh {wheel} {dest_dir}"`, because `delvewheel` is not cibuildwheel's Windows default. `PKG_CONFIG_PATH` is set per platform (`/project/.openblas` on Linux, `{project}` on macOS/Windows) so the S1 discovery step finds the CI-installed OpenBLAS inside the build container before repair ever runs.

**What transfers to Cytnx's OpenBLAS/ARPACK/hptt story.** Ch.03 §3.3 already established that Cytnx's Linux PyPI wheel today works only because `cibuildwheel`'s *default*, *undocumented-in-Cytnx's-own-docs* repair step happens to compensate for a missing `$ORIGIN` in `CMakeLists.txt:437-441` (ch.03 §3.2). SciPy's S2 shows the same underlying tool used the same way (no custom `repair-wheel-command` on Linux/macOS) but as *stated, deliberate build policy* in SciPy's own public docs — a packager reading `redistributable_binaries.html` is told exactly which tool runs and why, rather than having to reverse-engineer it from the absence of an override, which is what ch.02 §2.3(a) and ch.03 §3.3 had to do for Cytnx. S1 offers a second, independent lever for the same dependency class: instead of letting `find_library(ARPACK_LIB arpack REQUIRED)` (ch.03 §3.1 point 2) match whatever `libarpack.so` a build environment happens to expose, a SciPy-style pinned, pip-installable build-only artifact (the `scipy-openblasNN` pattern) would let Cytnx's CI resolve ARPACK/OpenBLAS from a version-pinned source instead of an ambient one — directly closing the "wrong lib found" gap ch.03 §3.2 failure mode 1 describes for ARPACK specifically.

## 4.2 CuPy — the CUDA-runtime and per-CUDA-version wheel playbook

### Pattern C1 — per-CUDA-major-version wheels instead of one fat/universal wheel

CuPy publishes a separate PyPI package per CUDA major version rather than a single package spanning all of them: `cupy-cuda12x` for CUDA 12.x and `cupy-cuda13x` for CUDA 13.x (current stable install docs, `https://docs.cupy.dev/en/stable/install.html`; the CuPy README, `https://github.com/cupy/cupy`, lists the same pattern plus `cupy-rocm-7-0` for the ROCm/AMD build). The `x` suffix means one wheel per package covers every *minor* release within that major version — i.e., the package-name axis is coarser than the CUDA version itself, matching CUDA's own within-major-version binary compatibility guarantee, while still drawing a hard line at major-version boundaries. This selection is resolved at CuPy's own build time, not left implicit: `setup.py` (`https://github.com/cupy/cupy/blob/main/setup.py`) computes `cuda_major = ctx.features["cuda"].get_version() // 1000` from the CUDA install detected during the build, and feeds it into an optional build dependency pinned to that exact major version, `f"cuda-toolkit[...]=={cuda_major}.*"` — the equivalent of Cytnx's `FindCUTENSOR.cmake:33-41` CUDA-major/minor branching (ch.02 §2.2), except CuPy surfaces the result as the package name itself rather than leaving it as an internal library-subdirectory choice invisible to the installer.

### Pattern C2 — locate the CUDA runtime through an ordered, documented search, without bundling the full toolkit

CuPy's install docs (`https://docs.cupy.dev/en/stable/install.html`) state the discovery order explicitly: "CuPy uses the first CUDA installation directory found by the following order" — (1) `cuda-pathfinder`'s own documented search order, (2) the `CUDA_PATH` environment variable, (3) the parent directory of an `nvcc` found on `PATH`, (4) the `/usr/local/cuda` default. This is the same *shape* of problem as Cytnx's `CUTENSOR_ROOT`/`CUQUANTUM_ROOT` handling (ch.03 §3.1 point 3, `CMakeLists.txt:180-186`), but structured differently: Cytnx hard-fails configuration if the env var is unset at all (`require_dependent_variable`), with no fallback search; CuPy treats the env var as one override among an ordered chain of fallbacks, so an unset `CUDA_PATH` degrades to a documented next step instead of a hard configure-time stop. The `cupy-cuda12x` PyPI package page (`https://pypi.org/project/cupy-cuda12x/`) states the corresponding runtime requirement directly: "This is a CuPy wheel (precompiled binary) package for CUDA 12.x. You need to install CUDA Toolkit 12.x locally to use these packages" — confirming the wheel itself does not bundle the full CUDA Toolkit. The same page documents an explicit opt-in alternative for users who don't want a system-wide toolkit at all: `pip install cupy-cuda12x[ctk]`, which pulls NVIDIA's own separately-published, per-component PyPI wheels (the current install docs name this the `[ctk]` extra, `https://docs.cupy.dev/en/stable/install.html`) so that, in that mode, only a compatible GPU driver — not a full local CUDA Toolkit install — is required.

### Pattern C3 — optional accelerators (cuTENSOR/NCCL/cuSPARSELt) as separate, independently-versioned add-on packages

CuPy's core wheel does not include cuTENSOR, NCCL, or cuSPARSELt. The install docs (`https://docs.cupy.dev/en/stable/install.html`) direct users to install them as their own CUDA-version-suffixed PyPI packages — `cutensor-cu13`, `nvidia-nccl-cu13`, etc. — or, on conda-forge, as ordinary optional packages (`conda install -c conda-forge cupy cutensor nccl`). CuPy previously shipped its own downloader for this purpose, `cupyx.tools.install_library` (`https://github.com/cupy/cupy/blob/main/cupyx/tools/install_library.py`); its own module docstring is "Installs the latest CUDA library supported by CuPy," and it fetches and unpacks prebuilt cuTENSOR/NCCL binaries for specific CUDA versions and architectures directly from NVIDIA's distribution — but the script is now marked deprecated in favor of installing these same libraries as ordinary versioned packages through pip or conda. The trajectory itself is the pattern: CuPy moved *away* from a bespoke, project-maintained fetch-and-unpack helper and *toward* treating each optional accelerator as a normal, independently-versioned package dependency, resolved by the same package manager (pip/conda) the user already used to install CuPy.

**Disclosure on scope (cuDNN/cuQuantum).** The install helper's own `--library` flag is hard-restricted to `choices=['cutensor', 'nccl']` (`install_library.py`, line 252 of the URL above) — cuSPARSELt is documented only as a separate pip/conda package on the current install docs and was never handled by this script at all. cuDNN *was* historically one of the libraries this same installer could fetch (a 2021 CuPy commit titled "Bump cuDNN to v8.1.0" modified `install_library.py`'s cuDNN support, `https://github.com/cupy/cupy/commit/0ff23221453ed3fcdf6146dfa5f838542b4abe29`), but it does not appear in the script's current `choices` list and is absent from the current install docs entirely — it was removed, not renamed or relocated. cuQuantum, by contrast, is not and has never been a CuPy-distributed accelerator: it appears nowhere in the CuPy repository or documentation checked for this chapter, and NVIDIA's own cuQuantum SDK page (`https://developer.nvidia.com/cuquantum-sdk`) describes it as "a standalone NVIDIA SDK, distributed independently — not through CuPy." The chapter's C3 label above therefore names CuPy's *actual* current optional-accelerator set (cuTENSOR/NCCL/cuSPARSELt) rather than the cuDNN/cuQuantum pairing a reader might expect from Cytnx's own accelerator story — the transferable pattern (optional GPU accelerators shipped as separate, independently-versioned opt-in packages rather than bundled into the core wheel) holds regardless of which specific libraries either project ships, which is why C3 still maps cleanly onto Cytnx's cuTENSOR/cuQuantum case below.

**What transfers to Cytnx's CUDA/cuTENSOR/cuQuantum story.** Ch.02 §2.3(b) and ch.03 §3.1 point 3 describe Cytnx's cuTENSOR/cuQuantum discovery as env-var-gated with no CI coverage at all (`USE_CUDA`/`USE_CUTENSOR`/`USE_CUQUANTUM` never built by any GitHub Actions workflow) and undermined by a dead-code `*_FOUND` check that always reports success (`FindCUTENSOR.cmake:65`, `FindCUQUANTUM.cmake:54`) regardless of whether `find_library` actually resolved anything — exactly the failure mode ch.03 §3.2 failure mode 3 (GPU driver/toolkit skew) describes concretely. C1 shows a way to make the CUDA-major-version coupling that already exists inside `FindCUTENSOR.cmake` an explicit, user-facing axis (a package name/variant) instead of an internal, silently-defaulting search-path choice. C2 shows that an env-var-driven search can degrade gracefully through an ordered fallback chain instead of refusing to configure at all, and that "don't bundle the whole toolkit" and "don't require a full manual toolkit install either" are not in tension — CuPy's `[ctk]` extra resolves that tension by delegating to NVIDIA's own per-component wheels rather than reinventing bundling logic in-project. C3 shows the direction of travel for an optional-accelerator story like Cytnx's cuTENSOR/cuQuantum: away from bespoke root-path environment variables validated by custom, bug-prone find modules, toward ordinary versioned package dependencies that a standard package manager — not project-specific CMake logic — resolves and can fail loudly on if absent.

## 4.3 Pattern → problem map

| Pattern | What it does | Chapter-03 problem it addresses |
|---|---|---|
| S1 — explicit, enumerated build-time BLAS/LAPACK selection (`-Dblas=`/`-Dlapack=`, pkg-config-then-CMake discovery, pinned `scipy-openblasNN` build dependency in CI) | Replaces ambient, unversioned discovery with a named build option resolved against a pinned, pip-installable artifact | §3.1 build-time discovery inconsistency (esp. point 1, `find_package`/`BLA_VENDOR` discovery with no pin) and §3.2 failure mode 1 (ARPACK "wrong lib found," name-only `find_library`) and failure mode 2 (LP64/ILP64 ABI mismatch — SciPy's `-Duse-g77-abi` gives an explicit override where Cytnx's `MKL_INTERFACE` default is silent) |
| S2 — post-build wheel repair (`auditwheel`/`delocate`/`delvewheel`) documented as deliberate, stated policy | Vendors the actually-linked shared libraries into the wheel and rewrites the loader path to find the bundled copy | §3.2 import-time resolution fragility generally, and specifically the RPATH gap ch.03 §3.2 identifies in Cytnx's own Linux `CMakeLists.txt:437-441` (no `$ORIGIN`) — SciPy shows the same compensating mechanism Cytnx already relies on (ch.03 §3.3), but named and documented rather than an undocumented side effect of not overriding `repair-wheel-command` |
| C1 — per-CUDA-major-version wheels (`cupy-cuda12x`, `cupy-cuda13x`) as an explicit package-selection axis | Surfaces the CUDA-major-version coupling that already exists internally as a user-facing, installer-resolved choice | §3.1 point 3 (env-var-gated cuTENSOR/cuQuantum discovery) and §3.2 failure mode 3 (GPU driver/toolkit skew) — makes the version coupling `FindCUTENSOR.cmake:33-41` already encodes internally into something the package manager, not silent internal branching, enforces |
