# 02 — Dependency Taxonomy

Reference commit: Cytnx `v1.1.0` (`8d96d928d22cc176615e97cafdb7a3cef66cb732`), the read-only submodule at `external/Cytnx`. As in chapter 01, citations are file paths and line numbers inside that submodule, or URLs to published package indices/license documents checked on 2026-07-03. This chapter profiles each native (non-Python) dependency Cytnx's C++ core and its `pycytnx` extension pull in, to establish which ones are easy to ship and which ones are structurally hard, before chapter 03 diagnoses the resulting problems.

## 2.2 Per-library profiles

### hptt

Role: accelerates tensor transpose/permutation operations in the native (non-`BACKEND_TORCH`) backend. Enabled by `USE_HPTT` (`cmake_dependent_option`, default `OFF` at the raw-CMake level but default `ON` in every `CMakePresets.json` preset via the shared `default` preset, `external/Cytnx/CMakePresets.json:14-20`).

Cytnx does **not** locate hptt with `find_package`. It is vendored as a git submodule (`thirdparty/hptt`, pointing at `https://github.com/Cytnx-dev/hptt.git`, `external/Cytnx/.gitmodules`) and built in-tree: `CytnxBKNDCMakeLists.cmake:64-68` `FATAL_ERROR`s if `thirdparty/hptt/CMakeLists.txt` is missing (i.e. the submodule wasn't checked out), then `add_subdirectory("${HPTT_SUBMODULE_DIR}" ...)` builds it as part of the same CMake configure pass (`external/Cytnx/CytnxBKNDCMakeLists.cmake:103`). The resulting `hptt_static` target is linked `PRIVATE` into `cytnx` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:278`) and additionally bundled into Cytnx's own `cytnx_targets` install-export set (`external/Cytnx/CytnxBKNDCMakeLists.cmake:300-301`) so that downstream `find_package(Cytnx)` consumers see `Cytnx::hptt_static` without needing their own `find_package(hptt)`.

Because it is statically linked in, hptt has **no independent runtime footprint** — whatever ships inside `libcytnx.a`/`pycytnx*.so` already contains it. This makes it categorically different from OpenBLAS/MKL/ARPACK: there is no separate `libhptt.so` a user or packager needs to locate on the target machine. Four codegen sub-options tune the build for specific hardware: `HPTT_ENABLE_ARM`, `HPTT_ENABLE_AVX`, `HPTT_ENABLE_IBM`, `HPTT_ENABLE_FINE_TUNE`, all default `OFF` and mutually exclusive with each other except `FINE_TUNE` (`external/Cytnx/CMakeLists.txt:171-175`). Only `HPTT_ENABLE_FINE_TUNE` is turned on by the shared `default` CMake preset (`external/Cytnx/CMakePresets.json:21`); none of `ARM`/`AVX`/`IBM` is set by any preset, including `openblas-apple`. **macOS/arm64 implication:** on Apple Silicon, hptt therefore compiles with generic (non-NEON-tuned) codegen unless a packager explicitly adds `-DHPTT_ENABLE_ARM=ON` — the preset that targets Apple hardware does not do this.

hptt's upstream (`springer13/hptt`) is BSD-3-Clause licensed. There is no PyPI package for hptt (`pypi.org/pypi/hptt/json` → 404, checked 2026-07-03) and a `hptt` package does exist on conda-forge (`api.anaconda.org/package/conda-forge/hptt` → 200, checked 2026-07-03), but Cytnx's build never consults it — the in-tree submodule is always used.

### OpenBLAS (with MKL as the alternative)

Role: default provider of BLAS, LAPACK, and LAPACKE — the linear-algebra kernels (matrix multiply, SVD, QR, eigendecomposition, etc.) underneath Cytnx's `Tensor`/`UniTensor` operations.

Selected when `USE_MKL=OFF` (the default, `external/Cytnx/CMakeLists.txt:169`). `CytnxBKNDCMakeLists.cmake:38-50` sets `BLA_VENDOR OpenBLAS`, then runs `find_package(BLAS REQUIRED)`, `find_package(LAPACK REQUIRED)`, and — only in this branch — `find_package(LAPACKE REQUIRED)` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:44`), because OpenBLAS bundles its own LAPACKE differently than a reference LAPACK install; the top-level `CMakeLists.txt` explains that Inria's `morse_cmake` submodule supplies the `FindLAPACKE` module used here specifically to "correctly handle LAPACKE bundled inside LAPACK (e.g. OpenBLAS)" (`external/Cytnx/CMakeLists.txt:13-19`). The found libraries are linked `PUBLIC` into `cytnx` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:45,55`).

OpenBLAS is a genuine **runtime** dependency: a shared `libopenblas.so`/`.dylib` must be resolvable at load time. No version is pinned by the CMake code. There is no ILP64/LP64 selection branch for OpenBLAS (unlike MKL, see below), implying the build assumes the standard 32-bit-integer (LP64) OpenBLAS interface. OpenBLAS is BSD-3-Clause licensed (conda-forge metadata, checked 2026-07-03), so redistribution is unrestricted. There is no single canonical "openblas" PyPI wheel usable as a link target for downstream builds (`pypi.org/pypi/openblas/json` → 404); Cytnx's own PyPI wheel instead bundles the shared library it was compiled against into the wheel itself, via `cibuildwheel`'s default repair tool (`auditwheel repair` on Linux, `delocate` on macOS — no custom `repair-wheel-command` overrides this in `[tool.cibuildwheel]`, `external/Cytnx/pyproject.toml:92-108`). CI installs the OpenBLAS headers/library before building via `dnf install ... openblas-devel` / `apk add openblas-dev` on Linux (`external/Cytnx/tools/cibuildwheel_before_all.sh`) and `brew install openblas` on macOS (`external/Cytnx/tools/cibuildwheel_before_all_macos.sh`). `openblas`/`libopenblas` packages exist on conda-forge (verified 200).

**macOS caveat (documented vs. actual):** `docs/source/adv_install.rst` warns that a plain Homebrew `openblas` install lacks the `lapacke.h` wrapper and recommends MKL on Mac instead (`external/Cytnx/docs/source/adv_install.rst:163-167`, already noted in ch.01 §1.1). Yet the actual macOS wheel-build config (`external/Cytnx/pyproject.toml:108`) installs and links Homebrew OpenBLAS directly via `CMAKE_PREFIX_PATH = "$(brew --prefix arpack):$(brew --prefix boost):$(brew --prefix libomp):$(brew --prefix openblas)"` with no MKL fallback — i.e., the documented workaround and the actual CI path diverge, consistent with ch.01's finding that `adv_install.rst` is stale relative to the real build.

**MKL (the alternative BLAS/LAPACK provider, `USE_MKL=ON`):** `CytnxBKNDCMakeLists.cmake:7-25` sets `BLA_VENDOR` to `Intel10_64ilp` or `Intel10_64lp` depending on the `MKL_INTERFACE` cache variable (default `"lp64"`, `external/Cytnx/CytnxBKNDCMakeLists.cmake:10,16-21`), reads `$ENV{MKLROOT}` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:13`), then runs the same `find_package(BLAS/LAPACK REQUIRED)` pair (no separate LAPACKE lookup — MKL ships its own). MKL is a **runtime** shared-library dependency exactly like OpenBLAS. Unlike OpenBLAS, MKL exposes an explicit ILP64-vs-LP64 integer-width choice via `MKL_INTERFACE`, a real ABI hazard if a downstream consumer's own MKL build uses the other width. MKL is proprietary, licensed under the "Intel Simplified Software License" (PyPI classifier for the `mkl` package, checked 2026-07-03), but both a runtime-only `mkl` and a headers+static `mkl-devel` package are published on PyPI by Intel (both verified 200) and on conda-forge (`mkl`, verified 200) — making MKL, despite being proprietary, one of the *easier* dependencies to obtain in prebuilt form. The conda recipe (`external/Cytnx/conda_build/meta.yaml:22-23,40-41,50-51`) selects `blas=*=mkl` on x86 and `blas=*=openblas` elsewhere; Cytnx's PyPI wheels, however, always build with the `openblas-cpu` preset regardless of platform (`external/Cytnx/pyproject.toml:77-80`), so **no PyPI wheel of Cytnx ever links MKL** — only the conda channel does, on x86.

### CUDA toolkit

Role: supplies the `nvcc`/CUDA-language toolchain and the runtime libraries (`cudart`, `cublas`, `cusparse`, `curand`, `cusolver`) that back the GPU tensor backend when `USE_CUDA=ON`.

Gated behind `USE_CUDA` (`cmake_dependent_option`, default `OFF`, `external/Cytnx/CMakeLists.txt:176`). When enabled, the top-level `CMakeLists.txt` first fixes `CMAKE_CUDA_ARCHITECTURES` to a fat-binary spanning `sm_70`, `sm_75`, `sm_80`, `sm_86`, `sm_89`, `sm_90`, plus `90-virtual` PTX if the user hasn't set architectures or `CUDAARCHS` (`external/Cytnx/CMakeLists.txt:209-210`), then calls `enable_language(CUDA)` (`external/Cytnx/CMakeLists.txt:212`). The actual dependency resolution happens in `CytnxBKNDCMakeLists.cmake:137-190`: `enable_language(CUDA)` again (idempotent) followed by `find_package(CUDAToolkit REQUIRED)` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:138`), then `target_link_libraries(cytnx PUBLIC CUDA::toolkit)` and `target_link_libraries(cytnx PUBLIC CUDA::cudart CUDA::cublas CUDA::cusparse CUDA::curand CUDA::cusolver)` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:189-190`) using CMake's modern imported `CUDA::*` targets, plus `-lcudadevrt` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:191`).

**Ambiguity flagged, not fabricated:** immediately below this (`external/Cytnx/CytnxBKNDCMakeLists.cmake:233-254`), there is a second block of `message(STATUS ...)` and `FILE(APPEND "${CMAKE_BINARY_DIR}/linkflags.tmp" ...)` calls referencing legacy-style variables (`CUDA_cusolver_LIBRARY`, `CUDA_curand_LIBRARY`, `CUDA_cublas_LIBRARY`, `CUDA_cudart_static_LIBRARY`, `CUDA_cudadevrt_LIBRARY`, `CUDA_cusparse_LIBRARY`) that are never `set()` anywhere near the modern `find_package(CUDAToolkit)`/`CUDA::*`-imported-target code path above it. These look like leftover code from an older `FindCUDA`-module-based build (pre-`CUDAToolkit`) that was not fully removed; they populate `linkflags.tmp` (used elsewhere for `pkg-config`-style export of raw flags) with empty/undefined variable expansions rather than failing the build, since CMake does not error on referencing an unset variable in this context. This does not affect the actual `target_link_libraries` link line above it, so it is not build-breaking, but it is a real piece of dead/stale code worth flagging rather than silently treating as intentional.

No CUDA toolkit version is pinned in CMake; `find_package(CUDAToolkit)` accepts whatever `nvcc`/CUDA install is discovered on `PATH`/`CUDA_HOME`. Because cuTENSOR's own discovery module branches on `CUDAToolkit_VERSION_MAJOR`/`MINOR` (see below), the effective CUDA version is load-bearing for GPU-stack compatibility even though Cytnx's own CMake never checks it.

CUDA is a genuine multi-library **runtime** dependency (`cudart`, `cublas`, `cusparse`, `curand`, `cusolver` are all separate shared objects). Redistribution of these specific runtime components is explicitly permitted under NVIDIA's CUDA Toolkit EULA (Attachment A lists `cudart`/`libcudart.so` and related libraries as redistributable, per `docs.nvidia.com/cuda/eula`, checked 2026-07-03) — this is the legal basis on which projects like PyTorch and JAX bundle CUDA runtime `.so` files inside their own PyPI wheels. NVIDIA itself publishes per-component PyPI wheels (e.g. `nvidia-cuda-runtime-cu12`, verified 200) that downstream projects can depend on. Cytnx does not use this pattern: as established in ch.01 §1.2, **no CI workflow builds `USE_CUDA=ON`**, so Cytnx's own PyPI wheels never link or bundle any CUDA library. A `cudatoolkit` meta-package exists on conda-forge (verified 200; license recorded as `LicenseRef-NVIDIA-End-User-License-Agreement`, confirming it is EULA-gated rather than freely redistributable at the package-manager level). **macOS/Windows:** NVIDIA has not shipped a macOS CUDA toolkit since CUDA 10.2; there is accordingly no `openblas-apple` + CUDA preset combination (ch.01 §1.1), and no Windows runner appears in any workflow, so CUDA is Linux-only in this codebase regardless of platform-support questions elsewhere.

### cuTENSOR

Role: GPU-accelerated tensor contraction and permutation kernels, used for tensor-network operations when `USE_CUTENSOR=ON`.

`USE_CUTENSOR` is a `cmake_dependent_option` on `USE_CUDA` (default `OFF`, `external/Cytnx/CMakeLists.txt:179`). Its location is **not** a standard `find_package` search path — it is required to come from the environment: `require_dependent_variable(CUTENSOR_ROOT "$ENV{CUTENSOR_ROOT}" PATH ...)` fails configuration with an explanatory message if `CUTENSOR_ROOT` isn't set whenever `USE_CUTENSOR` is on (`external/Cytnx/CMakeLists.txt:180-182`). The actual `find_package(CUTENSOR REQUIRED)` call (`external/Cytnx/CytnxBKNDCMakeLists.cmake:194`) resolves to Cytnx's own custom find module, `external/Cytnx/cmake/Modules/FindCUTENSOR.cmake`, which re-checks `CUTENSOR_ROOT`/`$ENV{CUTENSOR_ROOT}` and `FATAL_ERROR`s if neither is defined or the path doesn't exist (`external/Cytnx/cmake/Modules/FindCUTENSOR.cmake:17-27`).

The find module then branches the library subdirectory on the detected CUDA version: `CUDAToolkit_VERSION_MAJOR <= 10` → `10.2`; `11 <= major < 12` and `minor <= 0` → `11.0`; `11 <= major < 12` and `minor >= 1` → `11`; `major >= 12` → `12` (`external/Cytnx/cmake/Modules/FindCUTENSOR.cmake:33-41`), then does `find_library(CUTENSOR_LIB NAMES "cutensor" PATHS ${CUTENSOR_ROOT} PATH_SUFFIXES ${CUTNLIB_DIR} NO_DEFAULT_PATH)` and the same for `cutensorMg` (`external/Cytnx/cmake/Modules/FindCUTENSOR.cmake:47-60`). This is the core ABI hazard for cuTENSOR: **the cuTENSOR binary layout is versioned by the CUDA major/minor version it was built against**, so a `CUTENSOR_ROOT` pointing at a cuTENSOR install for CUDA 12 will not resolve correctly (or at all) against a CUDA 11 toolkit, and vice versa.

**Ambiguity flagged:** `external/Cytnx/cmake/Modules/FindCUTENSOR.cmake:65` unconditionally sets `CUTENSOR_FOUND TRUE` at the end of the module regardless of whether `find_library` actually located `cutensor`/`cutensorMg` (there is no check that `CUTENSOR_LIB`/`CUTENSORMg_LIB` are non-`NOTFOUND` before this). This means the `else()` branch in `CytnxBKNDCMakeLists.cmake:199-200` (`message(FATAL_ERROR "cannot find cutensor! please specify cutensor root with -DCUTENSOR_ROOT")`) is effectively dead code — `CUTENSOR_FOUND` is always true once `CUTENSOR_ROOT` itself resolves, so a wrong/incomplete cuTENSOR install would silently produce a link failure later (undefined references) rather than the intended clear configure-time error.

cuTENSOR is a **runtime** dependency: `libcutensor.so`/`libcutensorMg.so`, linked via `target_link_libraries(cytnx PUBLIC ${CUTENSOR_LIBRARIES})` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:198`). It is proprietary, under `LicenseRef-cuTENSOR-Software-License-Agreement` (conda-forge metadata, checked 2026-07-03). NVIDIA's cuTENSOR license supplement explicitly permits redistributing the runtime `.so`/`.h` files "as part of your application," subject to conditions (the application must have material functionality beyond the SDK itself, cannot expose the SDK as a standalone product, and must carry an NVIDIA attribution notice) — verified via `docs.nvidia.com/cuda/cutensor/latest/license.html`, checked 2026-07-03. NVIDIA publishes redistributable cuTENSOR packages on both PyPI (`cutensor`/`cutensor-cu12`, verified 200) and conda-forge (`cutensor`, verified 200). Cytnx itself doesn't use either channel — again, no CI workflow builds `USE_CUTENSOR=ON` (ch.01 §1.2). **macOS:** moot, since CUDA itself is unavailable.

### cuQuantum

Role: GPU-accelerated tensor-network contraction (`cuTensorNet`) and state-vector simulation (`cuStateVec`) primitives, used when `USE_CUQUANTUM=ON`.

Structurally identical to cuTENSOR: `USE_CUQUANTUM` is `cmake_dependent_option` on `USE_CUDA` (default `OFF`, `external/Cytnx/CMakeLists.txt:183`), and `require_dependent_variable(CUQUANTUM_ROOT "$ENV{CUQUANTUM_ROOT}" PATH ...)` fails configuration if `CUQUANTUM_ROOT` is unset when the option is on (`external/Cytnx/CMakeLists.txt:184-186`). `find_package(CUQUANTUM REQUIRED)` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:212`) resolves to `external/Cytnx/cmake/Modules/FindCUQUANTUM.cmake`, which re-validates `CUQUANTUM_ROOT`/`$ENV{CUQUANTUM_ROOT}` and `FATAL_ERROR`s otherwise (`external/Cytnx/cmake/Modules/FindCUQUANTUM.cmake:17-27`).

Unlike cuTENSOR, `FindCUQUANTUM.cmake` does **not** branch its library path on CUDA version — it hardcodes `set(CUTNLIB_DIR "lib")` unconditionally (`external/Cytnx/cmake/Modules/FindCUQUANTUM.cmake:30`) and searches `find_library(CUQUANTUM_TENSORNET_LIB NAMES "cutensornet" PATHS ${CUQUANTUM_ROOT} PATH_SUFFIXES lib NO_DEFAULT_PATH)` plus the equivalent for `custatevec` (`external/Cytnx/cmake/Modules/FindCUQUANTUM.cmake:36-49`). The same "always found" issue as cuTENSOR is present here too: `CUQUANTUM_FOUND` is unconditionally set `TRUE` at the end of the module (`external/Cytnx/cmake/Modules/FindCUQUANTUM.cmake:54`), regardless of whether the two `find_library` calls actually succeeded, so `CytnxBKNDCMakeLists.cmake:217-218`'s `FATAL_ERROR "cannot find cuquantum! ..."` branch is likewise effectively unreachable.

cuQuantum is a **runtime** dependency: `libcutensornet.so` and `libcustatevec.so`, linked via `target_link_libraries(cytnx PUBLIC ${CUQUANTUM_LIBRARIES})` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:216`). It is proprietary, recorded on conda-forge as `LicenseRef-cuQuantum-Software-License-Agreement` (checked 2026-07-03). **I could not verify the specific redistribution clause text for cuQuantum online** — NVIDIA's cuQuantum license documentation page returned only a table-of-contents/navigation shell rather than the license body when fetched, so I am not asserting it mirrors cuTENSOR's "bundle `.so`/`.h`" language, only noting that NVIDIA does publish redistributable cuQuantum artifacts on both PyPI (`cuquantum-python-cu12`, `custatevec-cu12`, `cutensornet-cu12`, all verified 200) and conda-forge (`cuquantum`, `cuquantum-python`, verified 200), which implies *some* redistribution grant exists even though its exact terms are unconfirmed here. As with cuTENSOR, no CI workflow in this repo builds `USE_CUQUANTUM=ON` (ch.01 §1.2), and it is unavailable on macOS for the same CUDA-absence reason.

### ARPACK

Role: iterative sparse-matrix eigensolver (implicitly restarted Lanczos/Arnoldi), used for large-scale eigenvalue problems (e.g. DMRG-style algorithms). Unlike every other library in this chapter, ARPACK is linked **unconditionally** — it is not gated by any `USE_*` option and is required even in `BACKEND_TORCH` builds.

Located with `find_library(ARPACK_LIB arpack REQUIRED)` (`external/Cytnx/CMakeLists.txt:307`), immediately after the Boost block and after the native-backend-vs-`BACKEND_TORCH` branch — i.e. it lives in the top-level `CMakeLists.txt`, not in `CytnxBKNDCMakeLists.cmake`, precisely because it applies to both backends. `find_library` here is a name-only search (no version or variant discrimination): it matches whatever library named `arpack` (`libarpack.so`/`.dylib`/`.a`) is visible on the linker's search path. The result is linked `PRIVATE` into `cytnx` (`external/Cytnx/CMakeLists.txt:309`).

ARPACK is a compiled **runtime** dependency — a genuine shared library must be present at load time (or a static `.a` at link time). Because `find_library` does no version check, and ARPACK's original reference implementation is Fortran77 with a name-mangling/calling-convention-sensitive ABI, mixing an ARPACK build from one Fortran compiler/runtime with a Cytnx build expecting another is a latent ABI hazard the CMake code does nothing to guard against. ARPACK (as packaged today, effectively "arpack-ng") is BSD-3-Clause licensed (conda-forge metadata, checked 2026-07-03). There is no standalone PyPI package providing a linkable ARPACK (`pypi.org/pypi/arpack/json` and `.../arpack-ng/json` both 404, checked 2026-07-03) — the only PyPI-adjacent presence of ARPACK is bundled inside SciPy's own `scipy.sparse.linalg` (`_arpack` submodule), which Cytnx does not use. Cytnx's own CI installs ARPACK via system/Homebrew package managers before compiling — `dnf install ... arpack-devel` / `apk add ... arpack-dev` on Linux (`external/Cytnx/tools/cibuildwheel_before_all.sh`), `brew install arpack` on macOS (`external/Cytnx/tools/cibuildwheel_before_all_macos.sh`) — and the resulting shared library is then bundled into the `cytnx` wheel by cibuildwheel's default repair step, the same mechanism used for OpenBLAS. A conda-forge `arpack` package exists (verified 200). **macOS arm64:** Homebrew's `arpack` formula covers both Intel and Apple Silicon, and the macOS build script installs it unconditionally alongside `boost`/`openblas`/`libomp`.

### pybind11

Role: generates the C++↔Python binding glue that becomes the `pycytnx` extension module, used only when `BUILD_PYTHON=ON` (default `ON`, `external/Cytnx/CMakeLists.txt:167`).

Discovery is two-tiered: `find_package(pybind11 3.0.0 QUIET)` first (`external/Cytnx/CMakeLists.txt:377`); if that fails (`NOT pybind11_FOUND`), CMake's `FetchContent` module downloads pybind11 from source — `FetchContent_Declare(pybind11_sources GIT_REPOSITORY https://github.com/pybind/pybind11.git GIT_TAG v3.0.1)` (`external/Cytnx/CMakeLists.txt:379-385`) — then `FetchContent_Populate` and `add_subdirectory` bring it into the build in-tree if not already populated (`external/Cytnx/CMakeLists.txt:387-396`). Either path feeds into `pybind11_add_module(pycytnx MODULE pybind/cytnx.cpp ...)` (`external/Cytnx/CMakeLists.txt:399`).

pybind11 is **header-only** — it emits no compiled runtime artifact of its own. Its entire contribution ends at compile time, when it expands into the templated glue code compiled directly into `pycytnx`'s own extension `.so`/`.pyd`. Consistent with this, `pyproject.toml` lists `pybind11 >=3.0` only under `[build-system].requires` (`external/Cytnx/pyproject.toml:1-3`) — a build-time-only pip dependency invoked by `pip`'s build isolation — never under `[project].dependencies`, so it is absent from the *installed* package's runtime dependency closure. The version is pinned two ways: `>=3.0.0` for the `find_package` path (`external/Cytnx/CMakeLists.txt:377`) and the exact tag `v3.0.1` for the `FetchContent` fallback (`external/Cytnx/CMakeLists.txt:384`). pybind11 is BSD-3-Clause licensed (confirmed by fetching its upstream `LICENSE` file), which is moot for redistribution purposes anyway since nothing of pybind11 itself ships in Cytnx's distributed artifacts. It is available both on PyPI (`pybind11`, verified 200 — ships headers + CMake config, not a binary wheel in the traditional sense) and conda-forge (`pybind11`, verified 200). No macOS/Windows-specific concerns beyond ordinary C++ header portability; being header-only, pybind11 imposes no per-platform packaging burden at all.

### Boost

Role: general-purpose C++ utility library. A source-level check of what Cytnx actually includes (`grep -rn "boost/" external/Cytnx/include external/Cytnx/src`) shows only header-only facilities: `boost/intrusive_ptr.hpp`, `boost/smart_ptr/intrusive_ptr.hpp`, `boost/checked_delete.hpp`, `boost/detail/atomic_count.hpp`, and `boost/unordered_map.hpp` — intrusive reference counting and an unordered-map implementation. None of Boost's compiled components (Boost.Filesystem, Boost.Serialization, Boost.Program_options, etc.) are used.

Discovery is `find_package(Boost REQUIRED)` with **no `COMPONENTS` argument** (`external/Cytnx/CMakeLists.txt:266`), and the result is linked as `target_link_libraries(cytnx PUBLIC Boost::boost ${LAPACK_LIBRARIES})` (`external/Cytnx/CMakeLists.txt:271`). `Boost::boost` is specifically CMake's `FindBoost` header-only interface target — it adds `Boost_INCLUDE_DIRS` to the include path (also done explicitly at `external/Cytnx/CMakeLists.txt:248,268-269`) and links **no** compiled Boost `.so`/`.a`. This is the concrete evidence for the brief's claim that Boost's shipping footprint depends on which components are used: in Cytnx's actual usage today, that footprint is exactly zero at runtime, because only header-only facilities are consumed and only the header-only target is linked.

Given that, Boost is **not** a runtime dependency for Cytnx as built today — no compiled Boost library appears in `cytnx`'s link line. This makes one packaging detail worth flagging: the conda recipe pins `boost >=1.82.0` under **both** `host:` (build-time, reasonable — headers are needed to compile) **and** `run:` (`external/Cytnx/conda_build/meta.yaml:39,47`), even though the actual CMake link graph never pulls in a compiled Boost library that would need to be present at runtime. This is either conservative packaging hygiene (protecting against a future compiled-component addition) or an unnecessary runtime pin — the CMake evidence alone doesn't resolve which, so this is noted as an observation rather than asserted as a bug.

No Boost version is enforced by the CMake `find_package(Boost REQUIRED)` call itself (no minimum version argument), unlike the conda recipe's separate `>=1.82.0` pin. Boost is licensed under the Boost Software License 1.0 (BSL-1.0, conda-forge metadata, checked 2026-07-03) — permissive, with no attribution or copyleft burden, which is largely moot here anyway since only headers are consumed. There is no PyPI package providing Boost's C++ headers: the literal package named `boost` on PyPI is an unrelated tool ("Boost automatically toots from Mastodon users," per its own PyPI summary, checked 2026-07-03), not the Boost C++ Libraries. Boost headers are obtained from system/Homebrew package managers in every CI path that builds Cytnx from source: `dnf install ... boost-devel` / `apk add ... boost-dev` on Linux (`external/Cytnx/tools/cibuildwheel_before_all.sh`), `brew install boost` on macOS (`external/Cytnx/tools/cibuildwheel_before_all_macos.sh`). A `boost`/`boost-cpp` package exists on conda-forge (verified 200). **macOS arm64:** Homebrew's `boost` formula builds for Apple Silicon; no special caveat surfaced beyond the general Homebrew-based install already covered above.

## 2.3 Classification

**(a) CPU math — OpenBLAS, MKL, ARPACK.** All three are compiled runtime shared libraries with real (if narrow) ABI surface: integer-width (LP64/ILP64) for the BLAS providers, Fortran calling-convention for ARPACK. None is version-pinned by Cytnx's CMake. All three, however, are permissively or commercially-but-redistributably licensed (BSD-3-Clause for OpenBLAS/ARPACK, Intel's redistributable "Simplified Software License" for MKL), are packaged on conda-forge, and — critically — are already being successfully bundled into Cytnx's own PyPI wheels today via `cibuildwheel`'s default `auditwheel`/`delocate` repair step after CI installs them from system package managers. This is the "solved" tier: the shipping mechanics work, they're just invisible (nothing in the docs or CMake output tells a packager this bundling is happening).

**(b) GPU stack — CUDA toolkit, cuTENSOR, cuQuantum.** This is the hard-to-ship tier, for four compounding reasons evidenced above: (1) all three are proprietary, NVIDIA-EULA-gated software — CUDA's own EULA only whitelists specific redistributable files (Attachment A), and cuTENSOR's/cuQuantum's redistribution terms live in separate, harder-to-verify license supplements; (2) cuTENSOR's library layout is explicitly version-coupled to the CUDA major/minor version it was built against (`FindCUTENSOR.cmake:33-41`), so a prebuilt artifact is tied to a specific CUDA runtime, not just "any CUDA"; (3) none of the three is reachable on macOS at all, since NVIDIA dropped macOS CUDA support after 10.2, so `openblas-apple` has no GPU counterpart; (4) per ch.01 §1.2, **no GitHub Actions workflow in this repository builds any of `USE_CUDA=ON`/`USE_CUTENSOR=ON`/`USE_CUQUANTUM=ON`** — meaning whatever shipping story exists for the GPU stack today is entirely manual and untested by CI, the single largest gap in this taxonomy. The two custom find modules (`FindCUTENSOR.cmake`, `FindCUQUANTUM.cmake`) also both have the same latent bug: `*_FOUND` is set unconditionally `TRUE` regardless of whether the `find_library` calls actually succeeded, so their intended `FATAL_ERROR` fallback in `CytnxBKNDCMakeLists.cmake` is dead code — a misconfigured `CUTENSOR_ROOT`/`CUQUANTUM_ROOT` fails at link time with a cryptic undefined-reference error instead of a clear configure-time message.

**(c) Build-only — hptt, pybind11, Boost (as used).** These three contribute nothing to Cytnx's runtime dependency closure. hptt is vendored and statically compiled in (no separate `libhptt.so` a packager must locate); pybind11 is header-only and, per `pyproject.toml`, explicitly scoped to build-time (`[build-system].requires`, never `[project].dependencies`); Boost, as Cytnx actually consumes it today, is linked only via the header-only `Boost::boost` interface target with zero compiled components. This tier is the easiest by construction — there is no runtime artifact to package, version-match, or redistribute.

**Overall ranking, hardest to ship first:** CUDA toolkit / cuTENSOR / cuQuantum (proprietary + CUDA-version-coupled + Linux-only + zero CI coverage) > OpenBLAS / MKL / ARPACK (compiled runtime libs, but permissive licensing and already successfully wheel-bundled) > hptt / pybind11 / Boost (build-time only, no runtime footprint). The GPU stack's difficulty is structural, not incidental: even if CI coverage were added, the proprietary redistribution terms and CUDA-version coupling would remain — this is the dependency cluster later chapters' channel evaluation (ch.05/06) will need to weigh most carefully.
