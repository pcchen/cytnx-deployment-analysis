# 01 — Current-State Audit

Reference commit: Cytnx `v1.1.0` (`8d96d928d22cc176615e97cafdb7a3cef66cb732`), pinned as the read-only submodule at `external/Cytnx`. All citations below are file paths and line numbers inside that submodule, or URLs to published artifacts checked on 2026-07-03.

## 1.1 Build system

### Entry points

- **`CMakeLists.txt`** is the top-level and only CMake entry point. It requires CMake ≥ 3.25 (`external/Cytnx/CMakeLists.txt:10`) — chosen specifically because 3.25 added CUDA device LTO via `INTERPROCEDURAL_OPTIMIZATION` (`external/Cytnx/CMakeLists.txt:10`). It fails fast with `FATAL_ERROR` if the `cmake_modules/morse_cmake` submodule (used for `FindLAPACKE`) has not been checked out (`external/Cytnx/CMakeLists.txt:16-18`), which means `git submodule update --init --recursive` is a hard prerequisite, not just a convenience step.
- It declares all `USE_*`/`BUILD_*`/`BACKEND_*` options (`external/Cytnx/CMakeLists.txt:167-191`), sets the version from `version.cmake` (`external/Cytnx/CMakeLists.txt:61-76`), creates the `cytnx` static library target (`external/Cytnx/CMakeLists.txt:233`), resolves Boost (`external/Cytnx/CMakeLists.txt:266`), branches into either the LibTorch backend or the native backend (`external/Cytnx/CMakeLists.txt:275-304`), links ARPACK unconditionally (`external/Cytnx/CMakeLists.txt:307-309`), and — if `BUILD_PYTHON=ON` — locates/fetches pybind11 and builds the Python extension (`external/Cytnx/CMakeLists.txt:375-390`).
- **`CytnxBKNDCMakeLists.cmake`** (302 lines) is `include()`-d only for the native (non-`BACKEND_TORCH`) backend (`external/Cytnx/CMakeLists.txt:303`). It is where the actual dependency-finding logic lives: BLAS/LAPACK/LAPACKE selection between MKL and OpenBLAS (`external/Cytnx/CytnxBKNDCMakeLists.cmake:6-58`), the HPTT in-tree submodule build (`external/Cytnx/CytnxBKNDCMakeLists.cmake:62-126`), and all CUDA/cuTENSOR/cuQuantum wiring (`external/Cytnx/CytnxBKNDCMakeLists.cmake:133-259`).
- **`version.cmake`** is a 3-line file that is the single source of truth for the version: `CYTNX_VERSION_MAJOR/MINOR/PATCH = 1/1/0` (`external/Cytnx/version.cmake:1-3`). `pyproject.toml` regexes this same file to derive the Python package version (`external/Cytnx/pyproject.toml:82-90`), so C++ and Python versions cannot drift independently. `CMakeLists.txt` additionally appends an optional PEP 440 dev suffix from the `CYTNX_VERSION_TAG` environment variable to form `CYTNX_VERSION_FULL`, used for nightly wheel versioning (`external/Cytnx/CMakeLists.txt:65-76`).
- **`CMakePresets.json`** (schema version 6, `external/Cytnx/CMakePresets.json:2`) defines the supported build configurations as named presets rather than ad hoc `-D` flags: `openblas-cpu`, `mkl-cpu`, `openblas-cuda`, `mkl-cuda`, `openblas-apple`, plus a `debug-*` variant of each (`external/Cytnx/CMakePresets.json:31-108`). The hidden `default` preset sets the baseline cache variables — including `BUILD_PYTHON=ON`, `USE_HPTT=ON`, `USE_CUDA=OFF`, `USE_MKL=OFF`, and `CMAKE_INSTALL_PREFIX=~/.local/cytnx` (`external/Cytnx/CMakePresets.json:14-29`). Only the `openblas-cuda`/`mkl-cuda` presets turn on `USE_CUDA`, `USE_CUTENSOR`, and `USE_CUQUANTUM` together (`external/Cytnx/CMakePresets.json:50-58`); there is no preset that enables CUDA without also enabling both cuTENSOR and cuQuantum, and no preset that builds `BACKEND_TORCH=ON`.

### The `USE_*` / `BUILD_*` / `BACKEND_*` option matrix

| Option | Default | Gates | How the dependency is located |
|---|---|---|---|
| `BUILD_PYTHON` | `ON` (`external/Cytnx/CMakeLists.txt:167`) | Builds the pybind11 Python extension module in addition to the C++ static library. | `find_package(Python COMPONENTS Interpreter Development)` then `find_package(pybind11 3.0.0 QUIET)`; if not found, `FetchContent` downloads pybind11 from source (`external/Cytnx/CMakeLists.txt:376-390`). |
| `BACKEND_TORCH` | `OFF` (`external/Cytnx/CMakeLists.txt:168`) | Switches the tensor storage backend to LibTorch instead of Cytnx's native CPU/CUDA backend; mutually exclusive with `USE_MKL`, `USE_HPTT`, and `USE_CUDA` (each is declared `cmake_dependent_option(... "NOT BACKEND_TORCH" OFF)`, `external/Cytnx/CMakeLists.txt:169,170,176`). | `execute_process` shells out to `python -c 'import torch; print(torch.utils.cmake_prefix_path)'` to locate the installed PyTorch's CMake config, then `find_package(Torch REQUIRED)` (`external/Cytnx/CMakeLists.txt:280-288`). No CMake option searches a user-specified Torch path directly — it depends on a working Python + `torch` import in the configuring environment. |
| `USE_MKL` | `OFF` (only meaningful when not `BACKEND_TORCH`) (`external/Cytnx/CMakeLists.txt:169`) | Selects Intel MKL vs. OpenBLAS as the BLAS/LAPACK provider. | If `ON`: sets `BLA_VENDOR` to `Intel10_64lp`/`Intel10_64ilp` per `MKL_INTERFACE`, reads `$ENV{MKLROOT}`, then `find_package(BLAS REQUIRED)` / `find_package(LAPACK REQUIRED)` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:7-35`). If `OFF`: `BLA_VENDOR=OpenBLAS`, then `find_package(BLAS/LAPACK/LAPACKE REQUIRED)` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:38-50`) — LAPACKE is only required in the OpenBLAS branch. |
| `USE_HPTT` | `OFF` (only meaningful when not `BACKEND_TORCH`) (`external/Cytnx/CMakeLists.txt:170`) | Accelerates tensor transpose using the bundled HPTT library. | Not `find_package`-based: HPTT is a git submodule at `thirdparty/hptt`; CMake `FATAL_ERROR`s if `thirdparty/hptt/CMakeLists.txt` is missing (`external/Cytnx/CytnxBKNDCMakeLists.cmake:63-68`) and otherwise `add_subdirectory`s it in-tree, bundling `hptt_static` into cytnx's own install export set (`external/Cytnx/CytnxBKNDCMakeLists.cmake:90-103,300-301`). Sub-options `HPTT_ENABLE_FINE_TUNE`, `HPTT_ENABLE_ARM`, `HPTT_ENABLE_AVX`, `HPTT_ENABLE_IBM` (all default `OFF`) further tune the codegen (`external/Cytnx/CMakeLists.txt:171-175`). |
| `USE_CUDA` | `OFF` (only meaningful when not `BACKEND_TORCH`) (`external/Cytnx/CMakeLists.txt:176`) | Enables the CUDA device backend (`UNI_GPU` compile definition), CUDA separable compilation, and (unless overridden) a fat binary spanning `sm_70` through `sm_90` plus `90-virtual` PTX for forward compatibility (`external/Cytnx/CMakeLists.txt:199-220`). | `enable_language(CUDA)` + `find_package(CUDAToolkit REQUIRED)`; links `CUDA::cudart/cublas/cusparse/curand/cusolver` (`external/Cytnx/CytnxBKNDCMakeLists.cmake:137-190`). |
| `USE_CUTENSOR` | `OFF`, `cmake_dependent_option` on `USE_CUDA` (`external/Cytnx/CMakeLists.txt:179`) | Enables cuTENSOR-accelerated tensor operations (`UNI_CUTENSOR`). | Custom find module `find_package(CUTENSOR REQUIRED)` (`external/Cytnx/cmake/Modules/FindCUTENSOR.cmake`), which requires `CUTENSOR_ROOT` to be set (env var or cache var) and `FATAL_ERROR`s otherwise (`external/Cytnx/CMakeLists.txt:180-182`, `external/Cytnx/cmake/Modules/FindCUTENSOR.cmake:16-27`). Library subdirectory selection is CUDA-major/minor-version-dependent. |
| `USE_CUQUANTUM` | `OFF`, `cmake_dependent_option` on `USE_CUDA` (`external/Cytnx/CMakeLists.txt:183`) | Enables cuQuantum-accelerated tensor-network primitives (`UNI_CUQUANTUM`). | Custom find module `find_package(CUQUANTUM REQUIRED)` (`external/Cytnx/cmake/Modules/FindCUQUANTUM.cmake`), which requires `CUQUANTUM_ROOT` and `FATAL_ERROR`s otherwise (`external/Cytnx/CMakeLists.txt:183-186`, `external/Cytnx/cmake/Modules/FindCUQUANTUM.cmake:16-25`). |

Other options exist but fall outside the scope of this matrix: `RUN_TESTS`, `RUN_BENCHMARKS`, `USE_DEBUG`, `BUILD_DOC`, `DEV_MODE` (all default `OFF`, `external/Cytnx/CMakeLists.txt:187-191`) control test/benchmark/sanitizer/doc builds rather than deployment-relevant dependencies. ARPACK is a required dependency regardless of any option (`find_library(ARPACK_LIB arpack REQUIRED)`, `external/Cytnx/CMakeLists.txt:307`), as is Boost (`external/Cytnx/CMakeLists.txt:266`).

Platform note: `CMAKE_INTERPROCEDURAL_OPTIMIZATION` (LTO) is force-disabled on Apple (`IF (APPLE) ... FALSE`, `external/Cytnx/CMakeLists.txt:227-231`) because it breaks the static-library link step on macOS; it stays `TRUE` on Linux/other platforms. `docs/source/adv_install.rst` separately documents a macOS-specific caveat: a plain Homebrew `openblas` lacks the `lapacke.h` wrapper, so MKL is recommended on Mac instead (`external/Cytnx/docs/source/adv_install.rst:163-167`). The `CMakePresets.json` `openblas-apple` preset (`external/Cytnx/CMakePresets.json:64-69`) has no CUDA-enabled counterpart, consistent with CUDA/cuTENSOR/cuQuantum being Linux+NVIDIA-only in this codebase.

## 1.3 Documented install paths for users

Three install paths are documented, all under `external/Cytnx/docs/source/`:

1. **Conda (recommended by the docs):** `docs/source/install.rst` tells users to `conda create --channel conda-forge --name cytnx python=3.9 _openmp_mutex=*=*_llvm` then `conda install -c kaihsinwu cytnx` (`external/Cytnx/docs/source/install.rst:21-22,58`), and for CUDA support, `conda install -c kaihsinwu cytnx_cuda` (`external/Cytnx/docs/source/install.rst:64`). The same file states explicitly: "We do not support a native Windows package at this stage. If you are using Windows OS, please use WSL." and "For MacOS: Please build from source, currently 0.9+ does not have conda package support." (`external/Cytnx/docs/source/install.rst:26,32`).
2. **`pip install .` from source (documented as "recommended for Python users"):** `docs/source/adv_install.rst` "Option C" describes `pip install .` or `pip install --editable .` from the repo root as the simplest path for Python-only users, driven by scikit-build-core (`external/Cytnx/docs/source/adv_install.rst:265-281`). This matches `pyproject.toml`'s `[build-system]` (`external/Cytnx/pyproject.toml:1-3`) and its pinned `--preset=openblas-cpu` default (`external/Cytnx/pyproject.toml:77-80`).
3. **CMake build-from-source (C++/advanced users):** `docs/source/adv_install.rst` "Option A" (cmake presets) and "Option B" (bare `cmake`/`make`/`make install`) walk through dependency installation (conda-forge or system package manager) and the full `-D` option table (`external/Cytnx/docs/source/adv_install.rst:1-263`). This is also the only documented path to a CUDA/cuTENSOR/cuQuantum build, via `conda install -c nvidia cuda`, `cutensor`, `cuquantum` plus `-DUSE_CUDA=on -DUSE_CUTENSOR=ON` (`external/Cytnx/docs/source/adv_install.rst:189-207`, `238`).

Note that `pip install cytnx` from PyPI directly (i.e. the published wheel, §1.2) is **not** mentioned anywhere in `docs/source/install.rst` or `docs/source/adv_install.rst`; only `docs/source/adv_install.rst`'s "Option C" (`pip install .`, a local source build) is documented, even though a prebuilt `cytnx` wheel has existed on PyPI since at least v1.1.0 (§1.2).

## 1.4 Gaps observed (facts only)

- **CUDA/cuTENSOR/cuQuantum builds have no CI coverage.** `CMakePresets.json` defines `openblas-cuda`, `mkl-cuda`, and their `debug-*` variants (`external/Cytnx/CMakePresets.json:50-63,95-103`), and `CMakeLists.txt`/`CytnxBKNDCMakeLists.cmake` fully implement `USE_CUDA`, `USE_CUTENSOR`, `USE_CUQUANTUM` (§1.1). No workflow file under `external/Cytnx/.github/workflows/` builds, tests, or publishes any artifact with `USE_CUDA=ON` (§1.2 cross-cutting observation). The only CUDA-build documentation that exists is the manual `adv_install.rst` walkthrough and its dedicated CUDA troubleshooting section on `nvlink`/`libnvvm.so` failures (`external/Cytnx/docs/source/adv_install.rst:344-390`), which itself demonstrates the CUDA path is exercised by humans, not CI.
- **`BACKEND_TORCH=ON` builds have no CI coverage.** The option and its full `find_package(Torch REQUIRED)` logic exist (`external/Cytnx/CMakeLists.txt:168,275-299`), but no `CMakePresets.json` preset sets `BACKEND_TORCH: "ON"`, and no workflow references it.
- **The docs advertise a `cytnx_cuda` conda package that no workflow builds.** `docs/source/install.rst:64` instructs users to run `conda install -c kaihsinwu cytnx_cuda`, but `conda_build/meta.yaml` always names the package `cytnx` (`external/Cytnx/conda_build/meta.yaml:5`) with no CUDA-enabled variant, and neither `conda_build.yml` nor `conda_build_release.yml` builds a CUDA configuration or a differently-named package.
- **The published conda package lags the pinned source version.** `anaconda.org/kaihsinwu/cytnx`'s `latest_version` was `1.0.0` at check time, one minor release behind the `v1.1.0` submodule commit that `conda_build_release.yml` is tag-triggered to build and upload (§1.2).
- **The macOS conda install path is documented as unsupported**, while `release_pypi.yml` does build and publish macOS wheels to PyPI for the same source tree (`external/Cytnx/.github/workflows/release_pypi.yml:123`) — the two distribution channels diverge in macOS coverage.
- **No Windows artifact of any kind is produced.** `external/Cytnx/conda_build/bld.bat` exists as a filename but is empty (0 lines), and no `windows-*` GitHub Actions runner appears in any workflow's `os:`/`runs-on` matrix. `docs/source/install.rst:26` documents this as a known limitation (WSL only).
- **`docs/source/adv_install.rst`'s raw-CMake option table is stale relative to `CMakeLists.txt`.** It lists a `-DUSE_OMP` option (`external/Cytnx/docs/source/adv_install.rst:235-236`) that does not exist anywhere in `external/Cytnx/CMakeLists.txt` or `external/Cytnx/CytnxBKNDCMakeLists.cmake`, omits `USE_CUTENSOR`, `USE_CUQUANTUM`, and `BACKEND_TORCH` entirely, and lists the `-DCMAKE_INSTALL_PREFIX` default as `/usr/local/cytnx` (`external/Cytnx/docs/source/adv_install.rst:228`) while `CMakeLists.txt` actually defaults it to `~/.local/cytnx` (`external/Cytnx/CMakeLists.txt:166`) and `CMakePresets.json`'s `default` preset also sets `~/.local/cytnx` (`external/Cytnx/CMakePresets.json:28`).
- **A CPU-only wheel is published on PyPI but not documented as an install path.** `docs/source/install.rst` and `docs/source/adv_install.rst` never mention `pip install cytnx` (the prebuilt wheel); the only pip instructions documented are `pip install .` / `pip install --editable .` against a local source checkout (§1.3), even though `release_pypi.yml` has published wheels to production PyPI on `v*` tags (§1.2).
