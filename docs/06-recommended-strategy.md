# 06 — Recommended Strategy & Roadmap

This chapter turns chapter 05's channel matrix into an actual recommendation for the Cytnx maintainers and a phased, issue-ready roadmap. It introduces no new evidence: every recommendation below traces to a finding already established in chapters 01–05, and each such trace is cited inline. Where a proposal goes beyond what chapters 01–05 directly establish, it is explicitly flagged as the author's own inference rather than a cited finding. Chapter 05 §5.4 is the load-bearing input — it concluded that the matrix "does not point to one channel that wins outright" but "points to a split between *what already works today* and *what the dependency shape structurally rewards*." This chapter accepts that split as the strategy rather than forcing a false single-channel answer.

## 6.1 Recommendation

**Ship a two-channel strategy, split by dependency tier, exactly as chapter 05 §5.4's matrix implies — do not pick one channel to do everything.** For the **CPU-only** configuration (OpenBLAS/ARPACK, ch.02 §2.3(a)), the recommended primary channel is **pip/wheels, with uv as a zero-cost drop-in on top**. Chapter 05 §5.4 found this is "the strongest channel for immediate, low-friction CPU access": it is already CI-built and published to PyPI today (ch.01 §1.2), its one covered configuration's runtime-resolution story is already solved by `cibuildwheel`'s default `auditwheel`/`delocate` repair (ch.02 §2.3(a), ch.03 §3.2), and its maintainer cost to keep working is near zero. uv "rides on exactly this same ceiling" (§5.4) — it adds resolver speed and a Python-level lockfile at no extra packaging cost. For the **GPU** configuration (CUDA/cuTENSOR/cuQuantum, ch.02 §2.3(b)) **and for users who need the most robust runtime resolution**, the recommended primary channel is the **conda-forge ecosystem — conda-forge itself, with pixi as an optional reproducibility layer on top**. §5.4 found conda(-forge)/pixi "score best on the two criteria the matrix weighs most heavily against Cytnx's actual dependency shape: runtime-resolution robustness and dependency coverage" — the entire proprietary GPU stack that pip/wheels cannot touch at all already exists as conda-forge packages (`cudatoolkit`/`cutensor`/`cuquantum`, ch.02 §2.1), and conda's solver structurally replaces the ambient-search-path fragility chapter 03 diagnosed (failure modes 1–2) with declared, version-tracked dependencies. **`build-from-source` is the fallback, not a channel** — §5.4 found nothing supports treating it as "more than the necessary baseline every other channel is built on top of"; it loses on robustness, reproducibility, and UX simultaneously (ch.05 §5.2), so it stays as the escape hatch for unsupported configurations, not a recommended install path.

Made explicit by platform, because chapter 05 §5.3 shows the split lands differently on each: **Linux + CUDA** has exactly one viable recommended home — **conda-forge** — because pip/wheels ships no CUDA wheel at all and never has (ch.01 §1.2), while conda-forge already carries the whole GPU stack (ch.02 §2.1); until a CUDA channel exists, Linux GPU users have no supported binary and must build from source (ch.05 §5.3 "Linux + CUDA"). **macOS (arm64 and x86_64)** stays on **pip/wheels**, which already builds and publishes working wheels on `macos-14`/`macos-15-intel` (ch.01 §1.2) with `delocate` handing a correct `@loader_path`-relative RPATH that Cytnx's own CMake already sets right on Apple (ch.03 §3.2) — CUDA is moot on macOS since NVIDIA dropped it after 10.2 (ch.02 §2.2). The one macOS opportunity the matrix surfaces (author's inference, extending ch.05 §5.3's observation): conda-forge could expose an Accelerate-linked BLAS build the way SciPy exposes `-Dblas=accelerate` as a named option (ch.04 §4.1 pattern S1), addressing the ch.03 §3.1 finding that Apple's own free, OS-provided Accelerate framework is absent from Cytnx's discovery logic entirely — but this is a later refinement, not a blocker for the recommendation above. Net: **pip/uv is the low-cost, already-working CPU story; conda-forge (with pixi on top) is where the GPU stack and the runtime-robustness problems chapters 02–03 diagnosed become structurally addressable** — the recommendation is to invest in both along the tier boundary the matrix already drew, not to collapse them into one channel.

## 6.2 The 2–3 highest-impact changes

Each is phrased so a maintainer could paste it into a GitHub issue. Effort estimates are the author's own rough sizing (S / M / L), not a cited finding.

1. **Fix the dead-code `*_FOUND TRUE` bug in the CUDA find-modules so a misconfigured GPU build fails loudly at configure time.** `external/Cytnx/cmake/Modules/FindCUTENSOR.cmake:65` and `FindCUQUANTUM.cmake:54` unconditionally set `CUTENSOR_FOUND`/`CUQUANTUM_FOUND` to `TRUE` regardless of whether their `find_library` calls actually located anything, which makes the intended `FATAL_ERROR` fallback in `CytnxBKNDCMakeLists.cmake:199-200`/`:217-218` unreachable dead code (ch.02 §2.2; ch.03 §3.1 ¶"Why this hurts reproducibility"). Consequence today: a wrong/incomplete cuTENSOR or cuQuantum install produces a build that "reports success" and fails later at link time or import time with no diagnostic (ch.03 §3.2 failure mode 3). Fix: gate each `*_FOUND TRUE` on the corresponding `find_library` result being non-`NOTFOUND`. This is a prerequisite for *any* trustworthy GPU packaging work (chapter 04 pattern C3 depends on discovery failing loudly, ch.04 §4.3). **Effort: S** (a few lines in two files).

2. **Add a relocatable `$ORIGIN` entry to the Linux `INSTALL_RPATH` branch and stop relying on unexpanded `~`.** `external/Cytnx/CMakeLists.txt:437-441` sets the non-Apple `INSTALL_RPATH` with no `$ORIGIN` token — the literal string `ORIGIN` appears nowhere in the tree — and includes the unexpanded literal `~/.local/cytnx/lib` from `CMAKE_INSTALL_PREFIX` (ch.03 §3.2 ¶"RPATH: what Cytnx's own CMake sets"), so Cytnx's own build does not produce a resolvable, relocatable RPATH on Linux; today only `cibuildwheel`'s default `auditwheel` repair compensates, and only for the one CPU config CI builds (ch.03 §3.2 ¶"What actually makes the Linux PyPI wheel work", ch.03 §3.3). The macOS branch already does this correctly with `@loader_path` (ch.03 §3.2), so this brings Linux to parity. This is the specific gap chapter 04's SciPy pattern S2 documents as the mechanism to make deliberate rather than accidental (ch.04 §4.3, S2 row). **Effort: S–M** (RPATH edit plus a check that the built extension carries a `$ORIGIN`-relative `RUNPATH`).

3. **Add CI coverage for at least one `USE_CUDA=ON` configuration.** No workflow under `external/Cytnx/.github/workflows/` ever builds `USE_CUDA=ON`/`USE_CUTENSOR=ON`/`USE_CUQUANTUM=ON` (ch.01 §1.2 cross-cutting observation; ch.02 §2.3(b)) — the single largest gap in the dependency taxonomy — so the entire GPU path is exercised only by humans, never validated, and no `auditwheel`/`delocate` repair ever runs against it (ch.03 §3.2 ¶"The gap this leaves for the GPU stack"). Without this, the conda-forge GPU recommendation in §6.1 has no automated proof it builds at all. Fix: add a workflow that configures an `openblas-cuda` preset (ch.01 §1.1) on a CUDA-capable runner or container and at minimum compiles and imports. This is the enabling step for chapter 04's CuPy patterns C1–C3 (ch.04 §4.2–4.3). **Effort: M–L** (new workflow plus sourcing a CUDA build environment in CI).

## 6.3 Phased roadmap

The three phases map onto chapter 02's dependency tiers and chapter 04's reference patterns. Each phase names the real Cytnx files it touches and states an acceptance signal.

### Phase 1 — Build-time discovery cleanup (make the build honest before packaging on top of it)

Rationale: chapter 03's central thesis is that import-time fragility is the runtime replay of build-time discovery ambiguity (ch.03 §3.3), so fixing discovery first is prerequisite to trusting any packaging layered above it.

Concrete steps and files:
- Fix the `*_FOUND TRUE` dead-code bug (§6.2 item 1) in `cmake/Modules/FindCUTENSOR.cmake` and `cmake/Modules/FindCUQUANTUM.cmake` (ch.02 §2.2, ch.03 §3.1).
- Add `$ORIGIN` to the Linux `INSTALL_RPATH` branch and remove the unexpanded `~` (§6.2 item 2) in `CMakeLists.txt:437-441` (ch.03 §3.2).
- Remove the stale legacy `CUDA_*_LIBRARY` / `linkflags.tmp` dead block in `CytnxBKNDCMakeLists.cmake:233-254`, left over from a pre-`CUDAToolkit` `FindCUDA` era (ch.02 §2.2 ¶"Ambiguity flagged, not fabricated").
- Correct the stale documentation chapters 01–02 flagged, so a source builder is not misled: the non-existent `-DUSE_OMP` option, the wrong `/usr/local/cytnx` install-prefix default, and the missing `USE_CUTENSOR`/`USE_CUQUANTUM`/`BACKEND_TORCH` options in `docs/source/adv_install.rst` (ch.01 §1.4), and the macOS OpenBLAS-vs-MKL documented-vs-actual divergence in the same file (ch.02 §2.2, ch.03 §3.1 "Platform angle").

Acceptance signal: configuring `openblas-cuda` with a `CUTENSOR_ROOT` that exists but contains no `libcutensor.so` now fails at configure time with the intended `FATAL_ERROR` (not at link/import time); the built Linux `cytnx*.so` carries a `$ORIGIN`-relative `RUNPATH`; `grep` for `USE_OMP` and `/usr/local/cytnx` in `adv_install.rst` returns nothing.

### Phase 2 — Packaging & wheels / conda-forge (harden and document the CPU story, close the channel-reality gaps)

Rationale: this is chapter 02's "solved" CPU tier (ch.02 §2.3(a)) — the mechanics already work but are invisible and the published channels drift from the source (ch.01 §1.4). Chapter 04's SciPy patterns S1/S2 are the playbook.

Concrete steps and files:
- Document `pip install cytnx` (the prebuilt PyPI wheel) as a first-class install path in `docs/source/install.rst` — today only `pip install .` from a source checkout is documented, though the wheel has shipped since v1.1.0 (ch.01 §1.3–1.4). Add uv (`uv pip install cytnx` / `uv add cytnx`) as the low-cost fast alternative (ch.05 §5.4).
- Adopt SciPy pattern S1: replace ambient `dnf/apk/brew`-installed OpenBLAS/ARPACK with a pinned, reproducible build-only source in `tools/cibuildwheel_before_all.sh` / `tools/cibuildwheel_before_all_macos.sh`, and document the `auditwheel`/`delocate` repair as deliberate policy (S2) rather than an undocumented side effect (ch.04 §4.1–4.3, closing ch.03 §3.2 failure mode 1's "wrong lib found" for ARPACK). Touches `pyproject.toml` `[tool.cibuildwheel]` and the two `before_all` scripts.
- Move the conda artifact from the personal `kaihsinwu` anaconda.org channel to a real **conda-forge feedstock**, and fix the version lag (anaconda.org shows `1.0.0` while the source is `v1.1.0`, ch.01 §1.2/§1.4). This makes the docs' already-stated `conda ... conda-forge` recommendation true (ch.01 §1.3, ch.03 §3.2 failure mode 4). Touches `conda_build/meta.yaml` (the existing CPU recipe is a real starting point, ch.05 §5.2 conda row) plus a new external feedstock repo.
- macOS refinement (author's inference from ch.05 §5.3): expose an Accelerate-linked BLAS build option in the conda recipe / CMake, mirroring SciPy's `-Dblas=accelerate` (ch.04 §4.1 S1), addressing the absent-Accelerate finding (ch.03 §3.1 "Platform angle"). Optional within this phase.

Acceptance signal: `pip install cytnx` and `uv add cytnx` are documented and succeed on Linux/macOS; a `cytnx` package installs from **conda-forge** (not `kaihsinwu`) at the current source version; the exact OpenBLAS/ARPACK build bundled into a wheel is pinned and reproducible build-to-build (ch.04 §4.1 S1's guarantee).

### Phase 3 — GPU accelerators (open the channel the matrix says is the only viable GPU home)

Rationale: chapter 02's "hard" tier (ch.02 §2.3(b)) — proprietary, CUDA-version-coupled, Linux-only, zero CI coverage. Chapter 04's CuPy patterns C1–C3 are the playbook; conda-forge is the recommended delivery channel (§6.1, ch.05 §5.4).

Concrete steps and files:
- Land §6.2 item 3: a new `.github/workflows/` file that builds and imports an `openblas-cuda` configuration on a CUDA-capable runner/container (ch.01 §1.1–1.2). This is the gate for everything else in this phase.
- Adopt CuPy pattern C2 in `cmake/Modules/FindCUTENSOR.cmake` / `FindCUQUANTUM.cmake`: turn the hard-fail `require_dependent_variable(CUTENSOR_ROOT ...)` (ch.03 §3.1 point 3) into an ordered search chain (env var → known default paths) that degrades gracefully instead of refusing to configure (ch.04 §4.2 C2, §4.3).
- Deliver GPU support through conda-forge by declaring `cudatoolkit`/`cutensor`/`cuquantum` as recipe dependencies (ch.02 §2.1) rather than env-var-gated root paths — this is CuPy pattern C3 (optional accelerators as normal versioned packages that fail loudly if unsatisfiable, ch.04 §4.2–4.3), and it lets conda's solver enforce the CUDA-major/minor coupling that `FindCUTENSOR.cmake:33-41` currently encodes silently (author's inference). Touches `conda_build/meta.yaml` and a CUDA-enabled recipe variant.
- If a pip/wheels GPU story is later wanted despite conda-forge being the primary GPU home: adopt CuPy pattern C1 — per-CUDA-major-version wheels (`cytnx-cuda12x`-style) making the version coupling a user-facing package-name axis (ch.04 §4.2 C1, §4.3). Flagged as optional/secondary because §6.1 places GPU on conda-forge, not pip.
- Reconcile the `cytnx_cuda` conda package the docs already advertise but no workflow builds (ch.01 §1.4) with whatever this phase actually ships.

Acceptance signal: CI builds, imports, and (ideally) tests a `USE_CUDA=ON` artifact on every relevant push; a Linux user can `conda install -c conda-forge cytnx` with GPU support and have cuTENSOR/cuQuantum resolved by the solver as declared dependencies; a deliberately-wrong CUDA/cuTENSOR version pairing fails at install/configure time, not silently at import (closing ch.03 §3.2 failure mode 3).

## 6.4 Follow-up phase (out of scope here)

Actually authoring and validating the concrete distribution configs — a working `pixi.toml`, a submitted conda-forge recipe/feedstock, and hardened pip/uv wheel CI — is deliberately a **follow-up** project and is **out of scope** for this analysis repository. This repo's job was to diagnose (chapters 01–03), study precedent (chapter 04), score the channels (chapter 05), and recommend (this chapter); it is not the place to build and ship the artifacts. Prototyping those configs requires things this repo intentionally does not have: write access to the upstream `Cytnx-dev/Cytnx` build files (here `external/Cytnx` is a read-only submodule, ch.01), CUDA-capable CI hardware to prove the Phase 3 GPU build (ch.01 §1.2), and the conda-forge feedstock review process (ch.05 §5.2's "real one-time-plus-ongoing cost not yet paid"). Each prototype should *validate* a specific claim this strategy rests on — a `pixi.toml` demonstrating the `pixi.lock` reproducibility guarantee §5.4 credits it with; a conda-forge recipe proving the GPU stack resolves through the solver (§6.3 Phase 3); a pip/uv wheel run proving the SciPy-S1 pinned-OpenBLAS build is reproducible (§6.3 Phase 2). Building and testing them belongs in the Cytnx source tree in that follow-up effort, not in this design-analysis repo — which is why no prototype config is committed here.
